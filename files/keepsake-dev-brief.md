# Keepsake — 开发简报 (Dev Brief for CC)

> 配套文件:`keepsake-full-app.html`（可交互原型，视觉 + 交互的唯一事实来源）。
> 本文档讲**做什么、为什么、怎么搭**；原型讲**长什么样、怎么交互**。两者一起看。
> 平台决策:**先 Web 后移动端**。技术栈按响应式 + 组件化选型,为日后移动端复用留路。

---

## 0. 一句话产品定义

**在对的时间，给对的人，发一封懂这段关系的邮件。**

- **邮件是主载体**（既是触达通道，也是表达本身）。
- **卡片是邮件里的一种可选形式**，不是产品核心，不能喧宾夺主。
- **关系驱动**:一切从“人”出发，不是从“卡片模板”出发。
- **核心壁垒**:① 懂关系（记得每个人的身份、节点、喜好忌讳）；② 懂东南亚多元文化（开斋节/屠妖节/农历新年的视觉、措辞、禁忌自动适配）。
- **首发市场**:东南亚，从马来西亚切入。**英文界面**。

---

## 1. 核心逻辑（务必先理解，再写代码）

### 1.1 关系 + 节点 → 决定邮件的主题与语气
这是整个产品的灵魂。同一个人，不同节点，邮件完全不同；同一个节点，不同关系，邮件也完全不同。

| 关系 | 节点 | 邮件语气 | 例 |
|---|---|---|---|
| 情人 | 纪念日 | 温柔亲密 / 暧昧 | “12 years, and still you” |
| 情人 | 对方生病 | 心疼慰问 | — |
| 朋友 | 开斋节 | 节庆问候（懂文化） | “Selamat Hari Raya, maaf zahir dan batin” |
| 长辈 | 生日 | 关切叮咛 | 提醒注意身体 |
| 朋友 | 久未联系 | 轻松想念 | “It's been too long” |

→ AI 写邮件时，输入必须包含 **关系类型 + 节点类型 + 这个人的档案（身份/文化/喜好/忌讳/历史互动）**，输出主题行 + 语气 + 正文。

### 1.2 文化自动适配（壁垒，不可省）
收件人的族群/信仰，决定节日、视觉、措辞、禁忌：
- **马来族 · 穆斯林** → Hari Raya；绿色系；“Selamat Hari Raya”；**禁忌**:不送圣诞祝福、礼物需 halal。
- **华人** → 农历新年；红色系（**禁用黑色**=丧）；“恭喜发财 / Gong Xi Fa Cai”。
- **印度裔 · 印度教** → Deepavali；暖橙灯火；“Happy Deepavali / Vanakkam”。

→ 维护一张 **文化规则表**（culture → {festivals, palette, greetings, taboos}）。AI 生成邮件/卡片时强制套用。这张表是产品资产，要可扩展（未来加新加坡、印尼等）。

### 1.3 不碰钱
**产品不做任何资金流**（无红包发钱、无代收代付）。原因:金融牌照（大马 Bank Negara / 新加坡 MAS）门槛太高。
- 实体卡片可**附带实体礼物**（节日糕点礼盒等）→ 走正常电商支付，赚商品分成。这是唯一涉及支付的部分，且是标准电商，非资金托管。

## 2. 信息架构（四页 + 个人中心）

参照原型左侧导航。五个顶层视图:

### ① Home — 提醒 + 最关心的人 + 主要关系
- 顶部:最该行动的节点（大卡片，一键进工作台写邮件）。
- 侧边:其他临近日期列表。
- 下方:重点关系（“closest circle”，加星标）网格。
- 作用:用户一进来就知道“现在该为谁做什么”。

### ② People（关系图谱）— 关系资产管理
- 按关系分组（Partner / Family / Friends / Colleagues）展示。
- 可**新增关系**。
- **点一个人 → 抽屉弹出完整档案**（关键页，AI 的数据地基）:
  - **Social identity**:关系类型 + 族群 + 信仰 + 备注标签
  - **Dates I'm watching**:需要关注的节假日/纪念日 + 距今天数
  - **What I know**:AI 了解的这个人（喜好、近况、忌讳）
  - **Mind these**:文化/个人禁忌（如“穆斯林，不送圣诞，礼物 halal”）
  - 底部:直接“为某节点写邮件”进工作台
- 关系类型用丰富下拉（18 预设跨亲密/家人/姻亲/朋友/职场/特定身份 + 自定义），每人打 关系 + 文化/信仰 标签。

### ③ Workspace（工作台）— 业务执行层，所有“做和发”在这
- 从某人某节点进入。**邮件为主体**:
  - AI 按 关系+节点 预设 **主题行 + 语气 + 正文**
  - 左侧 AI 助手:自然语言持续调整（更暧昧/加回忆/更短/换语气…）
  - 语气可一键切换（Tender / Playful / Heartfelt…）
  - **卡片作为可选附件**:挂在邮件正文下，可移除、可添加；卡片本身也由 AI 按人+节点+文化生成，支持插入用户照片并与生图融合
- 发送动作:**发邮件** / **寄实体卡片**（实体可加礼物）/ **定时**（到当天发）

### ④ History — 寄送历史
- 按月归档发过的邮件/卡片，带渠道（email/card）+ 状态（已送达/已读）。
- 作用:沉淀“守护了多久”的厚度，订阅粘性来源。

### ⑤ Profile（个人中心）
- **Sending**:邮件发送账号连接、寄信地址簿
- **Preferences**:提醒提前量、“My voice”（学习用户语气）、卡片默认风格
- **Account**:Keepsake+ 订阅、隐私与数据（加密，明确作为卖点）、登出

## 3. 技术栈建议（先 Web，留移动端路）

- **前端**:React + Next.js（App Router）+ TypeScript；Tailwind CSS（原型的设计 token 已可直接转 Tailwind 配置）。组件化拆分，便于日后 React Native / Expo 复用逻辑层。
- **状态/数据**:TanStack Query + 轻状态库（Zustand）。
- **后端**:Next.js API routes 或独立 Node 服务（NestJS 亦可）；优先快速验证。
- **数据库**:Postgres（关系数据天然适合）。核心表:`users / people / relationships / dates(nodes) / cultures / messages(emails) / cards / deliveries / subscriptions`。
- **鉴权**:邮箱 + OAuth（Google 起步，便于连邮件发送）。
- **隐私**:用户关系数据敏感，**静态加密**，最小权限。隐私是产品卖点，不是合规负担——在 Profile 里要能让用户看到“你的关系只属于你”。

## 4. 需要对接的外部能力（预留接口，别硬编死）

1. **LLM（邮件/卡片文案生成 + 对话调整）**
   - 输入:人档案 + 关系类型 + 节点 + 文化规则 + 历史互动 + 用户的对话指令
   - 输出:主题行 / 语气标签 / 正文（可流式）
   - “My voice”:用用户历史发送内容做轻量风格对齐（few-shot 或后续微调）。
2. **图像生成（卡片视觉）**
   - 预留 provider 接口（即梦/Seedream、Flux、GPT Image 等可切换）。
   - 能力:按 节日+文化+风格 生成卡面；**支持用户照片插入并融合**（匹配色调、留出文字位）。
3. **邮件发送**
   - 用用户连接的邮箱地址发送（保证“是用户本人发的”体感），或平台代发 + 用户域名。
   - 支持定时发送（到节点当天）。
4. **实体卡片打印 + 寄送履约**
   - 对接印刷/邮寄服务商（东南亚本地物流）。卡片导出为印刷规格 PDF。
   - 可选附带实体礼物（糕点礼盒）→ 标准电商下单 + 分成。
5. **提醒/调度**
   - 定时任务扫描即将到来的 nodes，按用户“提醒提前量”推送（邮件/站内/未来 push）。

## 5. 设计规范（从原型提取，务必遵守）

- **配色**:纯白主背景 + 天蓝主色 `#38A8F5`（深 `#2C90DE`，浅底 `#F0F8FE`）。左侧导航浅灰 `#F4F7FA`。蓝色只用于 logo/按钮/强调。
- **分区靠背景色块 + 留白，不用分隔线**。大量呼吸感、留白。
- **字体**:Inter（UI）+ Newsreader serif（情感/信件正文）。
- **图标**:全部 inline SVG symbol（原型已内置一套），**不要用图标字体**（线上加载会失败）。
- **头像**:首字母 initials + 彩色圆底。
- **风格基准**:Apple 级的克制、精致、国际化。无多余装饰，元素皆有功能。
- **文案**:杜绝营销腔（不要“赋能/落地/最/唯一”等）。第一人称从“收件人视角”出发，温度优先。

## 6. 建议的开发顺序（MVP 优先级）

1. **People 档案 + 文化规则表**（数据地基，先有这个 AI 才有料）
2. **Workspace 邮件生成 + 对话调整**（核心价值，邮件为主）
3. **邮件发送 + 提醒调度**（让闭环跑起来）
4. **Home 提醒仪表盘**（串起日常使用）
5. **卡片生成 + 作为邮件附件**（增强，含图像生成接口）
6. **History + Profile**（沉淀与设置）
7. **实体卡片寄送 + 礼物分成**（变现，可稍后）

> MVP 可以只做 邮件（含文化适配）+ 提醒 + 发送，先验证“懂关系的邮件”这一核心价值；卡片、实体、订阅随后叠加。

## 8. 一句话提醒 CC

> 不要把它做成“贺卡生成器”。它是“关系维护 + 懂文化的邮件”。卡片是配角，邮件和关系才是主角。任何时候 UI 让卡片抢了邮件的戏，就是跑偏了。
